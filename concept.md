# 역할 기반 AI 서비스 개발 개념

## 1. 핵심 정의

이 방식의 개발은 AI 앱 하나를 만드는 것이 아니다.

목표는 사용자가 원하는 기능과 역할을 정의하면, 브라우저, 데스크톱 앱, CLI, IDE 같은 여러 화면이 같은 AI 실행 구조를 통해 일을 맡길 수 있게 만드는 것이다.

```text
사용자 화면
- Chrome Extension
- Tauri Desktop App
- CLI
- IDE Extension
        ↓
작업 요청 계층
- 버튼
- 역할
- 프롬프트
- 입력 데이터
        ↓
Broker / Orchestrator
- 요청 정리
- 권한 확인
- 모델 연결
- 상태 관리
- 결과 스트리밍
        ↓
AI 실행 계층
- Codex
- ChatGPT
- Claude
- OpenAI API
- Local Model
        ↓
산출물
- 요약
- 번역
- 코드 수정
- 문서화
- 분석
- 자동화
```

핵심은 모델 자체가 아니라 `입력 -> 역할 -> 실행 -> 산출물 -> 기록` 흐름이다.

## 2. 개발 판단 순서

새로운 AI 기능이나 서비스를 만들 때는 모델부터 고르지 않는다.

항상 아래 순서로 판단한다.

```text
1. 사용자가 어디서 이 기능을 쓰는가?
2. 어떤 입력을 읽을 수 있는가?
3. AI에게 어떤 역할을 맡길 것인가?
4. 결과는 어떤 형태여야 하는가?
5. Broker가 필요한가?
6. 어떤 모델 또는 실행 엔진을 쓸 것인가?
7. 상태, 로그, 설정, 권한은 어디서 관리할 것인가?
8. 실패했을 때 사용자가 무엇을 봐야 하는가?
```

이 순서를 따르면 기능이 늘어나도 구조가 무너지지 않는다.

## 3. 사용자 화면 선택

AI 기능은 사용자가 실제로 일하는 위치에 붙어야 한다.

```text
웹페이지를 읽는 중이다 -> Chrome Extension
로컬 엔진을 관리한다 -> Tauri Desktop App
코드를 수정한다 -> IDE Extension
반복 작업을 실행한다 -> CLI / Cron
문서를 정리한다 -> Obsidian / Desktop App
```

좋은 AI 제품은 사용자가 데이터를 복사해서 가져오게 하지 않는다. 사용자가 이미 보고 있는 곳에서 입력을 읽고, 그 자리에서 결과를 돌려준다.

## 4. 역할 중심 설계

기능은 단순 버튼명이 아니라 역할로 정의하는 것이 좋다.

나쁜 정의:

```text
요약 버튼
번역 버튼
분석 버튼
```

좋은 정의:

```text
뉴스 해설자
기술문서 정리자
코드 리뷰어
투자 분석가
학습 코치
회의록 정리자
```

역할이 명확하면 프롬프트, 입력 데이터, 출력 형식, UI가 자연스럽게 결정된다.

예시:

```text
역할: 투자 분석가
입력: 현재 웹페이지 기사
출력: 사업 모델, 숫자, 리스크, 경쟁사, 투자 판단 포인트

역할: 개발 리드
입력: 기술 문서 또는 코드 diff
출력: 구현 포인트, 위험 요소, 필요한 작업 순서

역할: 학습 코치
입력: 어려운 문서
출력: 쉬운 설명, 비유, 핵심 개념, 복습 질문
```

## 5. 입력 데이터 설계

AI는 입력이 좋아야 일을 잘한다.

브라우저 입력:

```text
현재 탭 제목
URL
본문 텍스트
선택 영역
페이지 언어
메타데이터
```

로컬 프로젝트 입력:

```text
파일 내용
폴더 구조
git diff
에러 로그
테스트 결과
설정 파일
```

앱 상태 입력:

```text
사용자 선택값
이전 thread
프롬프트 설정
요청 기록
권한 상태
```

중요한 원칙:

```text
AI에게 모든 것을 던지지 말고, 작업에 필요한 입력만 정리해서 준다.
```

## 6. Broker가 필요한 경우

앱이 하나뿐이고 기능이 단순하면 직접 모델에 연결해도 된다.

하지만 아래 중 하나라도 해당하면 Broker가 필요하다.

```text
여러 앱에서 같은 AI 기능을 쓰고 싶다.
thread를 유지하고 싶다.
중간 진행 상황을 스트리밍해야 한다.
승인/거부 UI가 필요하다.
모델 선택이나 fallback이 필요하다.
요청 로그가 필요하다.
보안 정책이 필요하다.
나중에 자동화까지 확장하고 싶다.
```

Broker는 AI 서비스의 두뇌와 배전반이다.

```text
앱은 UX 담당
Broker는 정책과 실행 담당
모델은 추론 담당
```

## 7. 기본 아키텍처 패턴

```text
Chrome Extension
        ↓

AI Model
```

```text
웹페이지 요약

문서화
팩트 체크 보조

```

```text
Tauri App / CLI
        ↓
Local Broker
        ↓

        ↓
파일 시스템 / 터미널
```

적합한 기능:

```text
코드 수정

파일 정리

```

```text
        ↓

        ↓

        ↓

AI
        ↓

```

적합한 기능:

```text

학습 코칭

```

## 8. 출력 형식 설계

AI 결과는 그냥 텍스트가 아니라 사용 가능한 산출물이어야 한다.

예시:

```text
웹페이지 요약 -> 한 줄 결론 + 핵심 bullet
번역 -> 자연스러운 번역문 + 용어 메모
문서화 -> Markdown / HTML
코드 작업 -> diff + 변경 설명
리서치 -> 근거 + 판단 + 다음 행동
자동화 -> 상태 + 결과 + 실패 이유
```

출력 형식이 명확해야 UI도 명확해진다.

## 9. 상태와 신뢰성

AI 서비스는 사용자가 현재 상태를 믿을 수 있어야 한다.

보여줘야 하는 정보:

```text
Broker 연결 여부
AI 실행 계층 연결 여부
실제 사용 모델명
요청 진행 상태
마지막 확인 시각
오류 코드와 원인
```

특히 mock 응답과 실제 모델 응답은 반드시 구분해야 한다.

```text
모델: gpt-5.3-codex-spark
상태: 초록 / 노랑 / 빨강
```

## 10. 보안 원칙

개인 노트북에서 혼자 쓰더라도 보안 경계는 필요하다.

기본 원칙:

```text
Broker는 127.0.0.1에만 바인딩한다.
허용된 앱 origin만 Broker에 접근한다.
일반 웹사이트가 Broker에 직접 접근하지 못하게 한다.
Chrome Extension은 activeTab 중심으로 동작한다.
사용자가 클릭한 현재 탭만 읽는다.
비밀번호, 결제, 로그인 페이지는 자동 분석하지 않는다.
API key를 브라우저 확장에 넣지 않는다.
```

## 11. 로컬 모드와 API 모드

이 구조는 로컬 모드와 API 모드로 확장할 수 있다.

로컬 모드:

```text
개인 노트북
codex login
Codex app-server
Local Broker
Chrome Extension / Tauri App
```

API 모드:

```text
서버 배포
OpenAI API key 또는 별도 인증
사용자 계정
권한 관리
과금 관리
로그 관리
```

현재 우선순위는 로컬 개인용 모드를 안정화하는 것이다. API 모드는 제품 방향이 더 명확해진 뒤 분리해서 설계한다.

## 12. 개발 우선순위 원칙

새 기능을 만들 때 우선순위는 다음과 같다.

```text
1. 실제 사용 흐름이 맞는가?
2. 입력이 안정적으로 수집되는가?
3. 결과가 쓸 수 있는 형태인가?
4. 상태와 오류가 명확히 보이는가?
5. 보안 경계가 유지되는가?
6. 나중에 다른 앱에서도 재사용 가능한가?
```

## 13. 이 구조로 만들 수 있는 서비스 예시

```text
스마트 브라우저
- 현재 페이지 요약, 번역, 분석, 문서화

개발 리드 Assistant
- 코드 diff 리뷰, 리팩터링 계획, 구현 순서 제안

개인 리서치 Assistant
- 기사와 문서를 읽고 근거, 관점, 반론 정리

Obsidian 정리 Assistant
- 웹페이지나 대화 내용을 노트로 구조화

업무 자동화 Assistant
- 반복 작업을 버튼이나 cron으로 실행

학습 Assistant
- 어려운 문서를 강의 노트, 예제, 퀴즈로 변환
```

## 14. 최종 원칙

모델 중심으로 만들지 말고 작업 중심으로 만든다.

모델은 바뀔 수 있다.

```text
Codex
ChatGPT
Claude
OpenAI API
Local Model
```

하지만 좋은 작업 구조는 계속 유지된다.

```text
입력 -> 역할 -> 실행 -> 산출물 -> 기록
```

이 구조를 잘 만들면, 모델이 바뀌어도 서비스는 계속 확장할 수 있다.

최종 방향은 다음 문장으로 정리된다.

> 내가 원하는 기능과 역할을 정의하면, 브라우저와 로컬 앱이 그 역할을 실행 가능한 AI 작업으로 바꿔주는 개인용 AI 실행 플랫폼을 만든다.
